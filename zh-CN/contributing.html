贡献

本项目遵循 Read the Docs 行为规范。如果你不熟悉我们的行为准则政策，请在开始你的第一次贡献之前花一分钟时间阅读该政策。

修改主题

这个主题的样式使用 SASS 和一个名为 Wyrm 的自定义 CSS 框架。我们使用 Webpack 和 node-sass 来构建 CSS。Webpack 被用来观察变化，重建静态 assets，以及重建 Sphinx 演示文档。

注解

Node 的安装超出了本文档的范围。你需要 Node 版本 10 以上，才能对这个主题进行修改。

设置你的环境

  1. 安装 Sphinx 和文档构建依赖。

    $ pip install -e '.[dev]'

  2. 在本地安装 Webpack、node-sass 和主题依赖项。

    $ npm install

进行修改

对主题的修改可以用 Webpack 进行编译和测试：

$ npm run dev

这个脚本将做以下工作：

  1. 安装和更新任何依赖关系。

  2. 从 SASS 源文件建立静态 CSS。

  3. 建立演示文档。

  4. 注意 SASS 文件和文档的变更，在检测到的任何变更上重建一切。

另外，如果你不需要观察文件，可以使用发布构建脚本来测试构建的 assets：

$ npm run build

测试

由于向后兼容，QA 测试主题变更和拉动请求很复杂。

以下情况需要在改变 CSS 或 JavaScript 的情况下进行测试。

  • 应测试多种现代浏览器。我们目前正式支持回溯到 IE11

  • 应测试多种视口尺寸的变化。我们支持大尺寸、平板电脑和移动视口尺寸

  • 我们目前同时支持 Sphinx HTML4 和 HTML5 编写。这使得一些复杂的 CSS 选择器

  • 多个主要版本的 Sphinx 应该被测试。我们目前支持回溯到 Sphinx 版本 1.6

使用 tox 来测试依赖版本的组合是最简单的：

% tox -e py3-sphinx34-html4

如果测试和构建成功，你可以在 Sphinx 指出的目录中查看构建的文档：

build succeeded, 10 warnings.

The HTML pages are in .tox/py3-sphinx34-html4/tmp/html.
___________________________ summary ___________________________
  py3-sphinx34-html4: commands succeeded
  congratulations :)

然后你可以用一系列的浏览器打开这个路径进行测试。

发现用户界面问题的最好方法是比较两个或更多的构建。你可以建立多个 tox 环境，并将两者打开进行比较：

% tox -e py3-sphinx34-html4
...
% tox -e py3-sphinx34-html5
...
% firefox .tox/py3-sphinx34-html4/tmp/html/index.html
% firefox .tox/py3-sphinx34-html5/tmp/html/index.html

你也可以使用一个单独的 tox 环境来建立输出来进行比较。所有的 tox 环境都有一个额外的后缀 -qa，允许在不覆盖任何文件的情况下建立同一个环境两次。在这个测试场景中，你将从一个分支或标签开始构建，然后为你正在测试的拉动请求分支构建相同的 tox 环境。

例如，针对标签 0.5.2 进行测试：

% git checkout 0.5.2
% tox -e py3-sphinx34-html4-qa
...
% git checkout feature/example-pull-request
% tox -e py3-sphinx34-html4
...
% firefox .tox/py3-sphinx34-html4-qa/tmp/html/index.html
% firefox .tox/py3-sphinx34-html4/tmp/html/index.html

目前，对 QA 最重要的环境是：

  • py3-sphinx18-html4

  • py3-sphinx18-html5

  • py3-sphinx24-html4

  • py3-sphinx24-html5

  • py3-sphinx35-html4

  • py3-sphinx35-html5

  • py3-sphinx41-html4

  • py3-sphinx41-html5

翻译

翻译是使用 Transifex 管理的。你可以加入任何一个现有的语言团队，或者要求在项目中加入一个新的语言。关于我们的翻译标准的更多信息，请看我们在 Internationalization 的文档

定期地，核心团队应该在我们的正常发布之外更新翻译文件。核心团队中的某个人，如果有对 Transifex 的写入权限，应该运行以下程序：

$ python setup.py update_translations

这将提取新信息，将信息上传到 Transifex，并将更新我们的本地翻译文件。更改可以被检查到一个分支中，并被挂起供审查。

发布主题

为了发布新版本的主题，核心团队将采取以下步骤：

  1. 通过运行 bump2version [major|minor|patch|dev] 来提升版本。这将自动增加版本号的正确部分，你不需要指定准确的版本号。我们遵循 语义化版本 和 PEP440 （关于 alpha 版本和开发版本）。版本增量应该反映这些版本和任何潜在的破坏性变化。

  2. 新版本默认为 alpha 版本。如果这是一个候选版本，运行 bump2version --allow-dirty release 将该版本更新为 rc 版本。如果这是一个最终版本，请再次运行该命令。

  3. 用版本信息更新更新日志（docs/changelog.rst）。

  4. 运行 python setup.py update_translations 来编译新的翻译文件并更新 Transifex。

  5. 运行 python setup.py build_assets 来重建所有的主题 assets 和 Python 包。

  6. 提交变更。

  7. 在 git 中给该版本打上标签：git tag $NEW_VERSION。

  8. 推送标签到 GitHub：git push --tags origin。

  9. 将软件包上传到 PyPI：

    $ rm -rf dist/
$ python setup.py sdist bdist_wheel
$ twine upload --sign --identity security@readthedocs.org dist/*

  10. 最后，通过运行 bump2version patch 打开一个新的拉动请求，将开发版本更新到下一个补丁。用这一变更打开一个拉动请求。
